## 二、改进清单（按优先级排列）

### P0-1：Dropout 步恢复 `valid` 掩码（DeepSeek 发现）

**问题**：模态 Dropout 清零了先验，但 `valid` 掩码仍沿用"已知边=False"。这导致 20% 的训练步中，模型既没有先验辅助，又只被监督 ~50% 的候选边——**严格劣于原版 SAM-Road 的同类型训练步**。

**根因**：`sample_patch` 在 Dropout 之前执行，`valid` 掩码基于"有已知路网"的假设生成。Dropout 后先验被清零，但 `valid` 没有相应更新。

```
sample_patch() 计算 valid（已知边=False）
    ↓
__getitem__ 执行 Dropout: 清零先验，但 valid 不变
    ↓
20% 训练步：无先验 + 只有50%边被监督 = 双重劣势
```

**对比**：

| 训练步类型 | 先验信息 | 被监督边比例 | 步占比 |
|-----------|---------|-------------|--------|
| Completion 正常步 | 有 road_feat + GNN | ~50%（未知边） | 80% |
| **Completion Dropout 步（当前）** | **全空** | **~50%** | **20%** |
| 原版 SAM-Road | 全空 | 100% | 100% |
| **Completion Dropout 步（修复后）** | **全空** | **100%** | **20%** |

**方案**：在 `__getitem__` 的 dropout 分支中恢复 `valid` 为全 True。

```python
# data/dataset_completion.py, __getitem__ 中:
if drop_all:
    road_feature_map = np.zeros_like(road_feature_map)
    known_edge_index = torch.zeros(2, 0, dtype=torch.long)
    traj_heatmap = np.zeros_like(traj_heatmap)
    # ✅ 恢复 valid：让所有候选边都参与 loss（纯 extraction 模式）
    valid = tuple(
        tuple(True for _ in sample_valid) for sample_valid in valid
    )
```

**改动量**：~3 行  
**影响文件**：`data/dataset_completion.py`  
**风险**：极低。仅在 Dropout 步生效，正常步行为不变。

### P0-2：分割头改用 `fused_features`（Gemini 提出）

**问题**：分割头使用 `image_embeddings`（纯视觉），不受路网先验影响。当已知路段在卫星图上视觉不显著（遮挡/低对比度）时，分割头可能漏检 → mask 提取不出节点 → TopoNet 缺少关键节点 → GNN 无法传播已知拓扑 → 补全失败。

**级联失效链**：

```
已知路段视觉不显著 → 分割头漏检 → mask 无对应节点
  → TopoNet 缺少关键节点 → GNN 无法传播 → 补全失败
```

**有 traj 时的缓解**：4ch 输入时 traj 信号已经编码进 `image_embeddings`，分割头隐含了"哪里有路"的先验。但这对无 traj 场景（SpaceNet/CityScale）无效。

**方案**：分割头改用 `fused_features`。

```python
# models/sam_road_completion.py

# forward() 中:
# 旧: mask_logits = self.map_decoder(image_embeddings)
# 新:
mask_logits = self.map_decoder(fused_features)

# infer_masks_and_img_features() 中同理:
# 旧: mask_logits = self.map_decoder(image_embeddings)
# 新:
mask_logits = self.map_decoder(fused_features)
```

**防捷径措施**（同时实施，防止模型直接抄 road_feature_map 的 ch0）：

1. **mask loss 仍用完整 GT**（不区分已知/未知路段），模型必须对所有路段准确预测
2. **road_feature_map 随机遮挡增强**（见 P1-2），模拟先验缺失/错误
3. **保持模态 Dropout**

**消融实验**：

| 实验 | traj | 分割头特征源 | 预期效果 |
|------|------|------------|---------|
| A1 (基线) | 无 | image_embeddings | 已知路段可能漏检 |
| A2 (改进) | 无 | fused_features | 已知路段不漏检，无traj场景收益最大 |
| B1 (基线) | 有 | image_embeddings | traj已隐含先验，漏检风险较低 |
| B2 (改进) | 有 | fused_features | 双重先验注入，与B1差异应较小 |

**改动量**：~2 行（核心改动）  
**影响文件**：`models/sam_road_completion.py`  
**风险**：中等。可能过拟合已知路网，需配合防捷径措施。

### P1-1：增强无先验退化能力（Gemini 提出，DeepSeek 理论支撑）

**问题**：模型在 80% 训练时间中有先验，形成对先验的依赖。P0-1 修复后 Dropout 步等价于纯 Extraction，但 20% 比例可能不够——模型仍偏重"有先验"场景。

**DeepSeek 的理论支撑**：修正 Dropout 步后，模型的 Extraction 能力在理论上与 SAM-Road 对齐（100% 边监督），但实际效果仍取决于 Dropout 比例是否足够让模型在两种模式间充分切换。

**方案**：

```yaml
# config 改动:
MODALITY_DROPOUT_PROB: 0.3   # 20% → 30%, 增加无先验训练比例
```

**比例选择依据**：
- 太低（<20%）：退化能力不足，无先验时性能差
- 太高（>50%）：Completion 核心能力训练不充分
- 30% 是合理折中：70% 时间学习 Completion，30% 时间巩固 Extraction

**改动量**：1 行 config  
**影响文件**：config yaml  
**风险**：低。可通过实验调整比例。

### P1-0：三层 Dropout 策略的协同设计（DeepSeek 综合提出）

P0-1、P1-1、P1-2 三个改进并非独立修补，而应该作为**三层递进的 Dropout 策略**统一理解：

```
Layer 1 — 全量 Dropout (P1-1 调整后: 30%):
  road_feature_map = zeros, known_edge_index = empty, traj = zeros
  且 P0-1 修复: valid = 全 True
  → 模拟: 用户完全无先验 → 纯 Extraction，100% 边监督
  → 修复前: 20% 步，且只有 50% 边被监督（Bug）
  → 修复后: 30% 步，100% 边被监督（等价于原版 SAM-Road 训练步）

Layer 2 — 部分遮挡 (P1-2: 15%):
  road_feature_map = 随机块状遮挡
  known_edge_index = 保留
  traj = 按原有增强策略（全黑/腐蚀/完整）
  → 模拟: 用户先验不完整/有噪声 → 鲁棒 Completion
  → 关键: 必须在非 Layer1 步中独立采样（即 road_feat_erase_prob 不与 modality_dropout 重叠）

Layer 3 — 完整先验 (剩余: 55%):
  所有先验保持完整
  Xian 额外叠加 traj 增强策略（全黑20% + 腐蚀40% + 完整40%）
  → 模拟: 用户有高质量先验 → 最优 Completion
```

**三层分布**（P1-1 取 30% 时）：

| 层 | 先验状态 | valid 掩码 | 边监督比例 | 训练占比 | 模拟场景 |
|:--:|---------|-----------|-----------|:------:|------|
| 1 | 全空 | 全 True | 100% | 30% | 纯 Extraction |
| 2 | 部分遮挡 | 正常 | ~50% | ~10.5% | 低质量先验 |
| 3 | 完整 | 正常 | ~50% | ~59.5% | 高质量先验 |

**关键设计要点**：
1. Layer 1 和 Layer 2 必须**互斥**——`road_feat_erase_prob` 只在非 `drop_all` 步生效，且概率独立
2. Layer 1 的 valid 恢复是 P0-1 的核心贡献，使退化步从「无效训练」变为「有效 Extraction」
3. Layer 2 是 P0-2（分割头改特征源）的必要配套——如果分割头能看到 `fused_features`，必须同时训练它面对"融合特征不完全可靠"的情况

### P1-2：road_feature_map 随机遮挡增强（Gemini 提出）

**问题**：当前 road_feature_map 始终从精确 GT 坐标渲染，质量远高于推理时用户可能提供的路网。模型可能学到直接复制 road_feature_map ch0 的捷径。

**这是 P0-2 的必要配套措施**：如果分割头改用 `fused_features` 但不做遮挡增强，模型确实可能学到"road_feature_map ch0 有值 → 直接输出为路"的捷径。

**方案**：

```yaml
# config 新增:
ROAD_FEAT_ERASE_PROB: 0.15        # road_feature_map 随机遮挡概率
ROAD_FEAT_ERASE_REGIONS: [1, 3]   # 每次遮挡 1-3 个矩形区域
```

```python
# data/dataset_completion.py, __getitem__ 中:
if self.is_train and not drop_all:
    rand_val = random.random()
    if rand_val < self.road_feat_erase_prob:
        for ch in range(2):
            for _ in range(random.randint(*self.road_feat_erase_regions)):
                erase_w = random.randint(32, 128)
                erase_h = random.randint(32, 128)
                ex = random.randint(0, max(1, patch_size - erase_w))
                ey = random.randint(0, max(1, patch_size - erase_h))
                road_feature_map[ey:ey+erase_h, ex:ex+erase_w, ch] = 0
```

**与 traj 增强的对比**：

| 增强类型 | 目标 | 模拟场景 |
|---------|------|---------|
| traj 全黑 (20%) | 保持纯视觉能力 | 无轨迹 |
| traj 腐蚀 (40%) | 打断捷径 | 低质量轨迹 |
| road_feat 遮挡 (15%) | 防止复制捷径 | 已知路网部分区域缺失/错误 |

**改动量**：~15 行  
**影响文件**：`data/dataset_completion.py`、config yaml  
**风险**：低。不影响正常步，只在 15% 的训练步中生效。

### P1-3：训练/推理边映射对齐（Gemini 提出）

**问题**：训练时 `map_known_edges_to_nms` 硬性要求已知边端点精确在 NMS 节点集合中（可能丢边），推理时 `_match_known_edges_to_graph_points` 用 KDTree 最近邻（有距离误差但边保留率更高）。两者逻辑不一致，训练/推理存在 gap。

**DeepSeek 也指出了这个 gap**（§2.1 表格中 "精确 GT subdivided 图边索引" vs "KDTree 最近邻映射，可能误匹配"），但未给出具体方案。

**方案**：训练时也改用 KDTree 最近邻映射。

```python
# data/dataset_completion.py
def map_known_edges_to_nms(known_edge_set_subdivide, nmsed_indices,
                           patch_x0, patch_y0,
                           subdivide_points, nmsed_points,
                           distance_threshold=8.0):
    """
    将已知路网在 subdivided 图上的边映射到 NMS 后的节点索引
    优先精确匹配，失配时用 KDTree 最近邻（与推理时对齐）
    """
    if len(nmsed_indices) == 0 or len(known_edge_set_subdivide) == 0:
        return torch.zeros(2, 0, dtype=torch.long)

    # 1. 精确匹配映射
    sub_to_nms = {}
    for nms_idx, sub_idx in enumerate(nmsed_indices):
        sub_to_nms[sub_idx] = nms_idx

    # 2. 为未精确匹配的 subdivide 节点建立 KDTree 最近邻
    nms_kdtree = scipy.spatial.KDTree(nmsed_points)
    # nmsed_points 已经减去了 patch_x0/y0，subdivide_points 需要同样处理

    edges_src, edges_tgt = [], []
    for (s, t) in known_edge_set_subdivide:
        # 尝试精确匹配
        s_mapped = sub_to_nms.get(s, None)
        t_mapped = sub_to_nms.get(t, None)

        # 精确匹配失败，用 KDTree 最近邻
        if s_mapped is None:
            s_coord = subdivide_points[s] - np.array([patch_x0, patch_y0])
            dist, idx = nms_kdtree.query(s_coord)
            if dist < distance_threshold:
                s_mapped = idx
        if t_mapped is None:
            t_coord = subdivide_points[t] - np.array([patch_x0, patch_y0])
            dist, idx = nms_kdtree.query(t_coord)
            if dist < distance_threshold:
                t_mapped = idx

        if s_mapped is not None and t_mapped is not None and s_mapped != t_mapped:
            edges_src.append(s_mapped)
            edges_tgt.append(t_mapped)

    if len(edges_src) == 0:
        return torch.zeros(2, 0, dtype=torch.long)
    return torch.tensor([edges_src, edges_tgt], dtype=torch.long)
```

**改动量**：~30 行  
**影响文件**：`data/dataset_completion.py`  
**风险**：低。KDTree 最近邻是推理时已有的逻辑，移植到训练侧只是对齐。

### P2-1：推理时已知路网节点注入（Gemini 提出）

**问题**：推理时图节点完全由 mask 提取决定。如果已知路段在视觉上不显著，这些路段的节点可能无法被提取，导致 GNN 无法在对应位置传播已知拓扑信息。

**方案**：将已知路网的节点直接注入图节点集合。

```python
# engine/inferencer_completion.py, infer_one_img() 中:

# 1. 正常从 mask 提取节点
graph_points = extract_graph_points(fused_keypoint_mask, fused_road_mask, config)

if graph_points.shape[0] == 0 and known_graph_adj is None:
    return graph_points, np.zeros((0, 2), dtype=np.int32), fused_keypoint_mask, fused_road_mask

# 2. 注入已知路网节点 (去重)
if known_graph_adj is not None:
    known_nodes = set()
    for node, neighbors in known_graph_adj.items():
        known_nodes.add(node)
        for neighbor in neighbors:
            known_nodes.add(neighbor)

    if len(known_nodes) > 0:
        known_coords = np.array(list(known_nodes), dtype=np.float32)

        if len(graph_points) > 0:
            # 用 KDTree 检查已知节点是否已在 graph_points 中
            existing_kdtree = scipy.spatial.KDTree(graph_points)
            dists, _ = existing_kdtree.query(known_coords, k=1)
            # 距离 > NMS_RADIUS 的节点才需要注入（避免重复）
            inject_mask = dists > config.ROAD_NMS_RADIUS
            inject_coords = known_coords[inject_mask]
        else:
            # mask 完全提取不出节点，全部注入
            inject_coords = known_coords

        if len(inject_coords) > 0:
            graph_points = np.concatenate([graph_points, inject_coords], axis=0)
```

**注意事项**：
- 注入的节点没有 mask 提取的置信度，在 TopoNet 中可能引入噪声
- 注入节点的 `point_features` 仍从 `fused_features` 双线性采样，所以特征质量有保障
- 需要重新构建 `graph_rtree` 和 `known_edge_index_global`（注入后索引变了）

**改动量**：~30 行  
**影响文件**：`engine/inferencer_completion.py`  
**风险**：中等。注入噪声节点可能降低整体拓扑精度，需实验验证。

### P2-2：val 阶段双模式评估（Gemini 提出）

**问题**：当前 val 阶段保留 known_graph 但清空 traj，只评估"有先验"模式。无法衡量无先验退化程度，无法验证 P0-1/P1-1 的改进效果。

**方案**：val 阶段同时跑两种模式，分别记录指标。

```python
# models/sam_road_completion.py

def validation_step(self, batch, batch_idx):
    # 模式1: 有先验 (当前行为)
    loss_with_prior, metrics_with = self._compute_val(batch, drop_prior=False)
    self.log('val_loss', loss_with_prior, ...)
    self.log('val_road_iou', metrics_with['road_iou'], ...)

    # 模式2: 无先验 (每2个batch跑一次, 节省时间)
    if batch_idx % 2 == 0:
        loss_without_prior, metrics_without = self._compute_val(batch, drop_prior=True)
        self.log('val_loss_no_prior', loss_without_prior, ...)
        self.log('val_road_iou_no_prior', metrics_without['road_iou'], ...)

def _compute_val(self, batch, drop_prior=False):
    """计算一个 val step, drop_prior=True 时清空所有先验"""
    rgb = batch['rgb']
    traj_heatmap = batch.get('traj_heatmap', ...)
    road_feature_map = batch['road_feature_map']
    known_edge_index = batch.get('known_edge_index', None)

    if drop_prior:
        traj_heatmap = torch.zeros_like(traj_heatmap)
        road_feature_map = torch.zeros_like(road_feature_map)
        known_edge_index = None
        # 恢复 valid 掩码 (与 P0-1 同理)
        valid = tuple(tuple(True for _ in v) for v in batch['valid'])

    mask_logits, mask_scores, topo_logits, topo_scores = self(
        rgb, traj_heatmap, road_feature_map,
        graph_points, pairs, valid, known_edge_index
    )
    # ... 计算 loss 和指标
```

**改动量**：~30 行  
**影响文件**：`models/sam_road_completion.py`  
**风险**：低。不影响训练行为，仅增加 val 监控指标。

## 三、改进优先级总结

| 优先级 | 编号 | 改进 | 发现者 | 改动量 | 风险 | 理论收益 |
|--------|------|------|--------|--------|------|---------|
| **P0** | P0-1 | Dropout 步恢复 `valid` 掩码 | DeepSeek | ~3行 | 极低 | **修复结构性 Bug，20% 训练步从无效变有效** |
| **P0** | P0-2 | 分割头改用 `fused_features` | Gemini | ~2行 | 中等 | 消除级联失效链，已知路段不漏检 |
| **P1** | P1-0 | 三层 Dropout 策略协同设计 | DeepSeek | — | 低 | P0-1 + P1-1 + P1-2 的三层协同理解（非额外代码改动） |
| **P1** | P1-1 | 模态 Dropout 20%→30% | Gemini | 1行 | 低 | 增强无先验退化能力 |
| **P1** | P1-2 | road_feature_map 随机遮挡 | Gemini | ~15行 | 低 | P0-2 的防捷径配套，防止模型复制 road_feat |
| **P1** | P1-3 | 训练时边映射对齐推理时 | Gemini | ~30行 | 低 | 消除训练/推理 gap |
| **P2** | P2-1 | 推理时已知路网节点注入 | Gemini | ~30行 | 中等 | 保障 GNN 节点不缺失 |
| **P2** | P2-2 | val 双模式评估 | Gemini | ~30行 | 低 | 监控退化程度，验证改进效果 |

## 五、DeepSeek 未涉及的补充改进点

以下改进点由 Gemini 提出，DeepSeek 未涉及，经分析确认有价值：

### 5.1 4ch 第4通道对 image_embeddings 的影响分析

有 traj 时，`image_embeddings` 已经隐含了"哪里有路"的先验（通过第4通道注入 SAM Encoder）。所以：

- **有 traj + 分割头用 image_embeddings**：等效于分割头已经间接获得了路网先验，级联失效风险较低
- **无 traj + 分割头用 image_embeddings**：级联失效风险高，image_embeddings 完全不含路网先验

这意味着 P0-2（分割头改用 fused_features）的收益在**无 traj 场景更大**，有 traj 场景收益较小。消融实验应分开评估。

### 5.2 FeatureFusion 升级（低优先级）

当前 1×1 Conv 融合的局限性：无法建模"已知路网某条边如何影响远处未知区域"。但 GNN 路径B 已经通过消息传递部分弥补了这个局限。

**建议**：先在 P0-2 实验中观察 `fused_features` 的实际效果。如果发现"远处已知路网对补全影响不大"，则不需要升级 FeatureFusion。如果发现影响大但融合不足，再尝试 3×3 Conv 堆叠。

### 5.3 推理时已知路网节点注入

这是 P0-2 的推理侧补充。如果 P0-2 解决了"训练时分割头漏检"的问题，推理时 mask 质量应该提升。但如果 mask 仍然漏检（视觉遮挡严重），则 P2-1 节点注入可以作为保底方案。

**依赖关系**：P2-1 应在 P0-2 实验结果出来后再决定是否实施。
