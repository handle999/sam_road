# 实验编排方案设计 (Experiment Orchestration)

> 目标：把"4 任务 × 多数据集 × train+infer+eval"从 12+ 次手工命令，收敛为**一处声明配置、一条命令执行**，并支持单步和多步流水线、路径自动隔离、未来扩展数据集/脚本零成本接入。

> **状态：已实现并验证 (2026-06-23)**。本文档前半为设计，后半为实际用法。核心入口 `run.py`，注册表 `tools/registry.py`，统一目录 `runs/{run_id}/`。

## 快速开始 (已实现)

```bash
# 单个全流程: completion × spacenet (train+infer+eval)
python run.py --task completion --dataset spacenet --gpus 0

# 四任务一键批量 (batch.yaml 已配好: spacenet→GPU0, didi_xian→GPU1)
python run.py --batch batch.yaml --parallel

# 复用已有训练, 只重跑 infer+eval (ckpt 自动选 best)
python run.py --task completion --dataset spacenet \
    --run-id completion_spacenet_20260616_202527 --steps infer,eval --resume-run

# 只看不跑 (验证命令)
python run.py --task completion --dataset spacenet --dry-run
```

产物统一在 `runs/{run_id}/` 下，详见 [§四 统一目录树](#四统一目录树-runsid)。

## 一、现状痛点

当前要跑一个完整的"completion × spacenet"实验，需手工拼 3 条命令，且参数分布在 3 个地方：

```bash
# 1. train: 命令行 + config.yaml + 硬编码路径前缀 samroad_completion_{dataset}
python -m engine.train_completion --config config/..._completion.yaml --gpus 0

# 2. infer: 要手填 checkpoint 全路径、input_graph_dir、output_dir
CUDA_VISIBLE_DEVICES=0 python -m engine.inferencer_completion \
    --config config/..._completion.yaml \
    --checkpoint "checkpoints/samroad_completion_spacenet/completion-epoch=09-val_loss=0.1288.ckpt" \
    --input_graph_dir datasets/spacenet/RGB_1.0_meter

# 3. eval: 要手填 save 目录、dataset 名
cd metrics && python eval.py --dataset spacenet --dir save/infer_completion__20260618_... --workers 16
```

痛点：

| # | 痛点 | 根因 |
|---|------|------|
| 1 | 12 次命令太多，易错 | 三步解耦但无编排层串联 |
| 2 | 路径散落 4 套模式 | `samroad_` / `samroad_completion_` / `samroad_4ch_` / `infer__{ts}` 各自硬编码 |
| 3 | checkpoint 全路径手填 | 训练产出文件名含 `val_loss`，人工记不住 |
| 4 | 多步流水线无法表达 | "换模型重跑 infer+eval" 要重新拼两条命令 |
| 5 | 新增数据集/脚本成本高 | 路径前缀、partition、gt_pattern 散落在多个 inferencer 里 |
| 6 | `--gpus` 与 `CUDA_VISIBLE_DEVICES` 两套 GPU 指定方式并存 | train 用 `--gpus`，infer 用 env，eval 不用 |

---

## 二、设计目标

1. **一处声明**：所有实验参数在一个 yaml/profile 里声明，命令行只做覆写。
2. **一条命令**：`run.py` 统一入口，支持 `--steps train,infer,eval` 多步。
3. **路径自洽**：单一 `run_id` 贯穿 train→infer→eval，目录自动隔离、自动溯源。
4. **自主选择**：GPU / 数据集 / infer 用哪个 model / 哪个 ckpt 都可在命令行选。
5. **多步拆分**：既能 `train+infer+eval` 一条龙，也能 `infer+eval` 换模型重跑。
6. **零成本扩展**：新增数据集 = 加一个 dataset 注册项；新增脚本 = 加一个 task 注册项。

## 三、核心概念

### 3.1 三个正交维度，显式声明

| 维度 | 含义 | 示例 |
|------|------|------|
| **task** | 模型变体（决定用哪个 train/infer/test 脚本、哪个 model 类） | `extraction` / `completion` / `4ch` |
| **dataset** | 数据集（决定 config、partition、gt 路径、eval 参数） | `spacenet` / `didi_xian` / `cityscale` |
| **run_id** | 单次实验的唯一标识（贯穿三步） | `completion_spacenet_v1` |

一个**实验单元** = task × dataset × run_id。四任务即 `(extraction, spacenet)` `(extraction, didi_xian)` `(completion, spacenet)` `(completion, didi_xian)`，各起一个 run_id。

### 3.2 统一目录树（取代 4 套硬编码前缀）

所有产物归一到 `runs/{run_id}/` 下，按阶段分子目录：

```
runs/
  completion_spacenet_v1/                 # 一个 run_id 一个目录
    profile.yaml                          # 本次实验的完整声明快照 (含所有参数+git+env)
    run_info.yaml                         # 复用现有 dump_run_info 机制
    train/
      checkpoints/                        # *.ckpt (取代 checkpoints/samroad_completion_spacenet/)
        completion-epoch=09-val_loss=0.1288.ckpt
        last.ckpt
      log.txt                             # 取代 train_logs/samroad_completion_spacenet_<ts>.txt
      csv/                                # CSVLogger
    infer/
      graph/  mask/  viz/                 # 推理产物
      config.yaml  run_info.yaml  inference_time.txt
    eval/
      apls.json  topo.json
      apls/  topo/                        # 逐图明细
    best_ckpt.txt                         # 软链或路径文本, 指向 train/checkpoints 里 val_loss 最小的
```

**关键收益**：
- 路径隔离天然解决——不同 run_id 不同目录，绝无覆盖。
- 溯源完整——`profile.yaml` + `run_info.yaml` 记录 git/命令/env，任意一个产物都能反查怎么来的。
- 三步共享 `run_id`，infer 自动从 `runs/{run_id}/train/checkpoints/` 找 best ckpt，eval 自动读 `runs/{run_id}/infer/graph/`。

### 3.3 注册表机制（扩展性的核心）

把"task 用哪些脚本"和"dataset 有哪些属性"做成两张注册表，新增条目只需加一行。

```python
# tools/registry.py
TASKS = {
    'extraction': {
        'train_script':  'engine.train',
        'infer_script':  'engine.inferencer',
        'test_script':   'engine.test',
        'model_class':   'SAMRoad',
        'ckpt_prefix':   'epoch',
        'config_default': lambda ds: f'config/toponet_vitb_256_{ds}.yaml',
    },
    'completion': {
        'train_script':  'engine.train_completion',
        'infer_script':  'engine.inferencer_completion',
        'test_script':   'engine.test_completion',
        'model_class':   'SAMRoadCompletion',
        'ckpt_prefix':   'completion',
        'config_default': lambda ds: f'config/toponet_vitb_256_{ds}_completion.yaml',
    },
    # 未来新增 task: 4ch / contra / graph ... 加一项即可
}

DATASETS = {
    'spacenet': {
        'eval_dataset_name': 'spacenet',
        'infer_input_graph_dir': 'datasets/spacenet/RGB_1.0_meter',
        'traj_dir': None,                 # 无 traj
    },
    'didi_xian': {
        'eval_dataset_name': 'didi_xian',
        'infer_input_graph_dir': 'datasets/didi/xian/2019_400',
        'traj_dir': 'datasets/didi/xian/2019_400',  # active.png / region_{c}_traj.png 所在
    },
    # 未来新增数据集: 加一项 + 提供 partition/rgb_pattern 即可
}
```

**为什么这样能零成本扩展**：
- 新增数据集：在 `DATASETS` 加一项，把它的 `input_graph_dir`/`traj_dir`/`eval_dataset_name` 填上，run.py 自动用。inferencer 内部的 `if config.DATASET == 'spacenet'` 那套硬编码可逐步迁移到注册表（向后兼容）。
- 新增 task（比如新模型）：在 `TASKS` 加一项，指向新的 train/infer 脚本，run.py 自动编排。

## 四、统一入口 run.py

### 4.1 命令行接口

```bash
python run.py --task completion --dataset spacenet --run-id completion_spacenet_v1 \
    --gpus 0 --steps train,infer,eval
```

完整参数（分三组）：

```bash
# === 实验标识 (必填) ===
--task          TASK        extraction|completion|4ch|...  (注册表键)
--dataset       DATASET     spacenet|didi_xian|...          (注册表键)
--run-id        ID          本次实验唯一标识, 决定 runs/{id}/ 目录

# === 执行控制 ===
--steps         STEPS       train,infer,eval   (逗号分隔, 可只跑子集, 如 infer,eval)
--gpus          GPUS        "0" 或 "0,1"       (统一转 CUDA_VISIBLE_DEVICES + lightning devices)
--resume-run                 从 runs/{run-id} 续跑 (读 profile.yaml 恢复, 复用已完成的步)
--dry-run                    只打印将执行的命令, 不真跑

# === 覆写 (优先级: 命令行 > profile > config.yaml 默认) ===
--config        PATH        覆写 task 的默认 config
--checkpoint    PATH|auto   infer 用哪个 ckpt; "auto"=自动选 runs/{id}/train 里 val_loss 最小
--precision     16|32       训练/推理精度
--epochs        N           覆写 config.TRAIN_EPOCHS
--patience      N           early stopping
--infer-opts    "..."       透传给 inferencer 的额外参数 (如 --input_graph)
--eval-opts     "..."       透传给 eval.py 的额外参数 (如 --workers 16)
--device        cuda|cpu    推理设备
```

### 4.2 多步流水线语义

`--steps` 是核心，表达任意子流水线：

| 需求 | 命令 |
|------|------|
| 全流程 | `--steps train,infer,eval` |
| 只训练 | `--steps train` |
| 换模型重跑 infer+eval（复用已有 train） | `--steps infer,eval --checkpoint auto` 或指定新 ckpt |
| 只评估已有推理结果 | `--steps eval` |
| 训练断点续训 | `--steps train --resume-run` |

**步骤间依赖自动衔接**：
- `infer` 默认从 `runs/{run_id}/train/checkpoints/` 取 best ckpt（除非 `--checkpoint` 显式指定）。
- `eval` 默认读 `runs/{run_id}/infer/graph/`。
- `train` 默认输出到 `runs/{run_id}/train/`。
- 每步完成后写一个 `runs/{run_id}/.step_done/{step}` 标记，`--resume-run` 时跳过已完成步。

### 4.3 GPU 统一处理

`--gpus 0` 内部统一转成两套：
- 设置 `CUDA_VISIBLE_DEVICES=0`（inferencer/eval 用这个）
- 传给 lightning `devices=[0]`（train 用这个）

消除"`--gpus` vs `CUDA_VISIBLE_DEVICES`"并存的问题。

## 五、profile.yaml —— 单一声明源

每次 run 开始，把所有参数冻结成一份 `runs/{run_id}/profile.yaml`，三步都读它：

```yaml
# runs/completion_spacenet_v1/profile.yaml
run_id: completion_spacenet_v1
task: completion
dataset: spacenet

# 来源 (溯源用)
config_source: config/toponet_vitb_256_spacenet_completion.yaml
config_snapshot: |          # 把 config.yaml 内容内联, 保证可复现
  DATASET: spacenet
  SAM_VERSION: vit_b
  TRAIN_EPOCHS: 10
  ...

# 执行参数
steps: [train, infer, eval]
gpus: "0"
precision: 16
epochs: 10
patience: 0

# 衍生路径 (由 task+dataset+run_id 自动算出, 三步共享)
paths:
  run_root:        runs/completion_spacenet_v1
  train_dir:       runs/completion_spacenet_v1/train
  ckpt_dir:        runs/completion_spacenet_v1/train/checkpoints
  infer_dir:       runs/completion_spacenet_v1/infer
  eval_dir:        runs/completion_spacenet_v1/eval

# 步骤状态
step_status:
  train: pending
  infer: pending
  eval: pending

# 推理专属 (从 dataset 注册表 + 命令行覆写合成)
infer:
  checkpoint: auto            # auto = 训练后自动选 best
  input_graph_dir: datasets/spacenet/RGB_1.0_meter
  traj_dir: null
  device: cuda
  extra_opts: ""

# 评估专属
eval:
  metric: all
  workers: 16

# 环境/git 快照 (run_info 机制复用)
git: {commit: ..., branch: main, dirty: false}
env: {cuda_visible_devices: "0", ...}
created_at: 2026-06-22T...
```

**好处**：任何一个 `runs/{run_id}/` 都是一个**自包含、可复现**的实验包——把目录拷给别人，照着 `profile.yaml` + `config_snapshot` 就能还原。

## 六、四个任务的一键声明

四个实验单元，各写一行 `--run-id` 即可，或用 batch 配置文件批量提交：

```bash
# 单条
python run.py --task extraction  --dataset spacenet  --run-id extraction_spacenet_v1  --steps train,infer,eval --gpus 0
python run.py --task extraction  --dataset didi_xian --run-id extraction_xian_v1     --steps train,infer,eval --gpus 1
python run.py --task completion  --dataset spacenet  --run-id completion_spacenet_v1 --steps train,infer,eval --gpus 0
python run.py --task completion  --dataset didi_xian --run-id completion_xian_v1     --steps train,infer,eval --gpus 1
```

或一个 `batch.yaml`：

```yaml
# batch.yaml — 一次声明, 顺序/并行执行
runs:
  - {task: extraction, dataset: spacenet,  run_id: extraction_spacenet_v1, gpus: "0"}
  - {task: extraction, dataset: didi_xian, run_id: extraction_xian_v1,     gpus: "1"}
  - {task: completion, dataset: spacenet,  run_id: completion_spacenet_v1, gpus: "0"}
  - {task: completion, dataset: didi_xian, run_id: completion_xian_v1,     gpus: "1"}
default_steps: [train, infer, eval]
```

```bash
python run.py --batch batch.yaml           # 顺序跑全部
python run.py --batch batch.yaml --parallel # 每个分配到对应 GPU 并行 (受 --gpus 约束)
```

## 七、与现有代码的兼容策略（关键：渐进迁移，不破坏现有工作流）

### 7.1 不动现有脚本的核心逻辑

`run.py` **不重写** train/infer/eval，而是**作为编排层调用**它们，通过环境变量 + 命令行参数注入路径：

- 训练：`run.py` 调 `python -m engine.train_completion --config ... --gpus ...`，但通过新增的 `--run-root` 参数让 train 脚本把 ckpt/log 写到 `runs/{id}/train/` 而非硬编码的 `checkpoints/...`。
- 推理：同理注入 `--output_dir runs/{id}/infer`、`--checkpoint auto-resolved`、`--input_graph_dir`。
- 评估：调 `metrics/eval.py --dir runs/{id}/infer --dataset ...`。

### 7.2 最小侵入改动点

给每个 train/infer 脚本加一个**可选**参数 `--run-root`：

```python
# train_completion.py 增加 (向后兼容, 不给则走老路径):
parser.add_argument("--run-root", default=None,
    help="若提供, ckpt/log 写到 {run-root}/train/ 下; 否则走默认 checkpoints/ 路径")
...
if args.run_root:
    ckpt_dir = f"{args.run_root}/train/checkpoints"
    log_path = f"{args.run_root}/train/log.txt"
else:
    ckpt_dir = f"checkpoints/samroad_completion_{dataset_name}/"   # 老路径不变
    ...
```

inferencer 同理加 `--run-root`，替代 `--output_dir` 的拼接逻辑。**不给 `--run-root` 时行为完全不变**，老命令照常工作。

### 7.3 best ckpt 自动选择

`run.py` 在 train 后、infer 前扫描 `runs/{id}/train/checkpoints/`，按文件名里的 `val_loss=` 数值选最小的，写入 `runs/{id}/best_ckpt.txt`，infer 时读它。无需人工记 checkpoint 全名。

### 7.4 eval 入口已统一

`metrics/eval.py` 已经是统一入口（`--dataset --dir --metric --workers`），`run.py` 直接调，无需改 eval。

## 八、多步拆分的典型场景

| 场景 | 命令 | 说明 |
|------|------|------|
| 首次全流程 | `--steps train,infer,eval` | 三步串行，路径自动衔接 |
| 训练完了，调阈值后重跑 infer+eval | `--steps infer,eval --resume-run` | 复用 train 产物，不重训 |
| 换 ckpt 重跑 infer+eval | `--steps infer,eval --checkpoint runs/.../train/checkpoints/last.ckpt` | 指定任意 ckpt |
| 只想看新指标 | `--steps eval` | 读已有 infer 结果 |
| 训练中断续训 | `--steps train --resume-run` | 跳过已完成步，读 last.ckpt 续 |
| 并行对比两模型 | 两个 `--run-id` 不同 GPU | 互不干扰 |

**断点恢复**靠 `runs/{id}/.step_done/` 标记 + `last.ckpt`，`--resume-run` 自动续。

## 十、实现路线（分阶段，建议顺序）

| 阶段 | 内容 | 价值 | 改动量 |
|------|------|------|--------|
| **P0** | `tools/registry.py`（TASKS/DATASETS 两表） | 扩展基础 | 新文件 |
| **P0** | `run.py` 单步执行（先支持 `--steps infer` 或 `eval`，调通编排） | 验证可行性 | 新文件 |

| **P1** | `run.py` 支持 `train,infer,eval` 多步 + best ckpt 自动选 | 主流程可用 | run.py 扩展 |
| **P1** | `profile.yaml` 生成 + `--resume-run` | 可复现可续跑 | run.py 扩展 |
| **P2** | `--batch batch.yaml` 批量 + `--parallel` | 四任务一键 | run.py 扩展 |

## 十一、与现有 run_info 机制的关系

现有 `tools/run_info.py` 的 `dump_run_info` / `mark_run_finished` **完全复用**：
- `profile.yaml` 是**实验级**声明（人写的意图 + 冻结参数）
- `run_info.yaml` 是**步骤级**执行记录（自动抓 git/env/command/duration）
- 两者互补：profile 说"我想做什么"，run_info 说"实际怎么跑的、跑了多久"

每个 `runs/{id}/` 下：1 个 `profile.yaml` + 每步 1 个 `run_info.yaml`（train/infer 各一份）。

## 十二、总结

**一句话**：引入 `run.py` 编排层 + `runs/{run_id}/` 统一目录树 + 两张注册表（TASKS/DATASETS），把 4 套硬编码路径收敛为单一 `run_id` 贯穿的三步流水线，命令行只做声明与覆写，新增数据集/脚本只需加注册表一项。

**核心收益**：
1. 12 次命令 → 4 条（或 1 个 batch.yaml）
2. 4 套路径模式 → 1 套 `runs/{id}/` 树
3. checkpoint 手填 → 自动选 best
4. 多步流水线 → `--steps` 表达任意子集 + `--resume-run` 续跑
5. 扩展 → 注册表加一行，run.py 零改动
6. 可复现 → 每个实验目录自包含 profile + run_info

**设计原则**：编排层不重写底层脚本，只通过可选的 `--run-root` 参数注入路径，老命令完全不受影响，可渐进迁移。

## 十三、实际实现说明 (2026-06-23 已落地)

### 13.1 落地的文件清单

| 文件 | 作用 |
|------|------|
| `run.py` | 编排入口，单步/多步/batch/resume/dry-run |
| `tools/registry.py` | TASKS + DATASETS 两张注册表，路径计算 |
| `tools/config_utils.py` | run_root 路径构造、profile 读写、best ckpt 选择、step 标记 |
| `tools/migrate_ckpts_to_runs.py` | 一次性迁移脚本，把旧 checkpoints/ 迁到 runs/ |
| `batch.yaml` | 四任务批量示例 |
| `engine/train.py` `train_completion.py` `train_4ch.py` | 加 `--run-root`，路径走 runs/{id}/train/ |

### 13.2 与原设计的差异

1. **目录策略改为彻底迁移**（用户决策）：废弃 `checkpoints/` `save/`，所有脚本统一写 `runs/{id}/`。旧 ckpt 已迁移并补 profile，可用 `--resume-run` 复用。
2. **run_id 默认 `{task}_{dataset}_{timestamp}`**（用户决策）：每次唯一不冲突，可 `--run-id` 覆写。
3. **数据集命名统一为 `didi_xian`**：config 文件从 `_xian` 重命名为 `_didi_xian`，代码里 `('xian','didi_xian')` 兼容写法清理为单一 `didi_xian`。注册表里 dataset key 即 config 文件名用词，无需特殊映射。
4. **config 内联快照**：profile.yaml 内联完整 config 内容（`config_snapshot` 字段），runs/{id}/ 拷到别处可还原。
5. **batch + parallel P0 就做**：`batch.yaml` + `--parallel` 一开始就支持，四任务可并行（按各自 GPU）。

### 13.4 步骤间依赖自动衔接

- `infer` 默认 `--checkpoint auto`：自动从 `runs/{id}/train/checkpoints/` 选 val_loss 最小的 ckpt（写入 `best_ckpt.txt`）。
- `eval` 自动读 `runs/{id}/infer/graph/`。
- `train` 输出到 `runs/{id}/train/`。
- 每步完成写 `runs/{id}/.step_done/{step}`，`--resume-run` 跳过已完成步。
