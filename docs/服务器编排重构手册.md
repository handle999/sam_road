# 服务器编排重构手册 (Server Orchestration Refactor Playbook)

> **用途**：让服务器上的 Claude（或任何人）按本手册，把一个**已有自己权重和 .gitignore 的 sam_road 仓库**重构为统一的 `run.py` 编排体系。
>
> **核心原则**：服务器环境和本地不同——它有自己训练的权重（不同时间戳、可能不同 epoch）、自己的 .gitignore、可能在 `checkpoints/` `save/` `outputs/` 里有产物。**所有步骤必须先探测现状再操作，绝不照搬本地的时间戳或假设**。
>
> **前置条件**：本仓库已经合并了编排体系的代码（`run.py` / `tools/registry.py` / `tools/config_utils.py` / 6 个脚本已加 `--run-root`）。如果服务器仓库还没有这些文件，先 `git pull`。本手册只处理**产物迁移与环境适配**，不重新实现代码。

在动手前，Claude 必须先跑完这个自检，搞清楚服务器现状。**把每一步的输出记录下来**，后面的操作都基于这些事实。

```bash

  engine/inferencer.py engine/inferencer_completion.py engine/inferencer_4ch.py

```

### 0.2 探测现有产物（关键！不要假设）

```bash
# 1. 现有权重在哪？什么命名？
echo "=== checkpoints/ ==="
find checkpoints -name "*.ckpt" 2>/dev/null | sort
echo "=== save/ (推理产物) ==="
ls -d save/*/ 2>/dev/null | head

ls outputs/ 2>/dev/null

# 2. 现有 train_logs
echo "=== train_logs/ 文本日志 ==="

# 4. 现有 .gitignore（服务器可能自己改过）
echo "=== .gitignore 关键行 ==="

```

**记录下**：服务器有几个 ckpt、在哪些目录、文件名里的 `val_loss` 和 `epoch` 是什么。这些决定了迁移映射。

### 0.3 确认 git 状态

```bash

```

**若 `git status` 有服务器自己的未提交改动**：先理解这些改动是什么（可能就是服务器自己训练产物的配置），决定是否需要先 commit/stash，再继续。

## 一、理解目标结构（重构后的样子）

重构后所有产物归一到 `runs/{run_id}/`，`run_id = {task}_{dataset}_{timestamp}`：

```
runs/
  completion_spacenet_20260616_202527/      # 一个 run_id 一个目录
    profile.yaml          # 实验声明快照 (进 git, 溯源)
    best_ckpt.txt         # best ckpt 路径 (进 git)
    .step_done/train      # 步骤完成标记 (进 git)
    train/
      checkpoints/        # *.ckpt (不进 git, *.ckpt 全局忽略)
      log.txt             # 训练文本日志 (不进 git)
      run_info_*.yaml     # 步骤级执行记录 (进 git)
      csv/                # CSVLogger (不进 git)
    infer/
      graph/ mask/ viz/   # 推理产物 (不进 git)
      config.yaml run_info.yaml inference_time.txt
    eval/
      apls.json topo.json # (json 进 git)
      apls/ topo/         # 逐图明细 (不进 git)
```

**核心收益**：`run.py --task completion --dataset spacenet --gpus 0` 一条命令跑完 train+infer+eval，ckpt 自动选 best，路径自动隔离。

## 二、迁移现有权重到 runs/ 体系

### 2.1 服务器权重的两种情况

服务器上的权重可能来自两种训练：

**情况 A：用旧脚本训练的权重**（在 `checkpoints/samroad_*/` 下，文件名含 `val_loss`）
→ 需要用迁移脚本搬到 `runs/`。

**情况 B：已经用新 `run.py --run-root` 训练的权重**（已经在 `runs/{id}/train/checkpoints/`）
→ 无需迁移，跳到第三节。

先判断：

```bash
if [ -d checkpoints ] && [ "$(find checkpoints -name '*.ckpt' 2>/dev/null | head -1)" ]; then
  echo "情况 A: 旧权重在 checkpoints/, 需迁移"
elif [ -d runs ] && [ "$(find runs -name '*.ckpt' 2>/dev/null | head -1)" ]; then
  echo "情况 B: 已在 runs/, 无需迁移"
else
  echo "无现有权重, 跳过迁移 (后续用 run.py 直接训练)"
fi
```

### 2.2 迁移脚本的工作原理（先理解再执行）

`tools/migrate_ckpts_to_runs.py` 的逻辑：

1. 扫描 `checkpoints/` 下的子目录，目录名规则：
   - `samroad_{dataset}` → task=extraction
   - `samroad_completion_{dataset}` → task=completion
   - `samroad_4ch_{dataset}` → task=4ch
   - `samroad_completion`（无后缀，历史遗留）→ task=completion, dataset=spacenet
2. 从目录里的 `run_info_*.yaml` 提取时间戳，拼出 `run_id = {task}_{dataset}_{ts}`
3. 把 `.ckpt` 和 `run_info` 移到 `runs/{run_id}/train/` 下
4. 读 `run_info` 的 `config_source`，补建 `profile.yaml`（含 config 内联快照）
5. 标记 train 步骤完成

**服务器权重的 dataset 可能与本地不同**（比如服务器训了 cityscale，本地没有）。迁移脚本是数据驱动的，会自动识别目录名里的 dataset，不需要改脚本。

### 2.3 执行迁移（先 dry-run 预览）

```bash
# 第一步：dry-run，看清楚会做什么
python tools/migrate_ckpts_to_runs.py --dry-run
```

**仔细核对 dry-run 输出**：
- 每个旧目录 → `runs/{task}_{dataset}_{ts}/` 的映射对不对？
- task 识别对不对？（`samroad_spacenet` → extraction，`samroad_completion_spacenet` → completion）
- dataset 识别对不对？
- ckpt 文件都列出来了吗？

**如果 dry-run 显示"跳过无法识别的目录"**：说明服务器有非标准命名的目录。手动检查：
```bash
ls checkpoints/   # 看那个无法识别的目录名
```
如果是服务器自定义的命名（比如 `my_exp_spacenet`），你需要手动决定它的 task/dataset，然后手动迁移（见 2.4）。

### 2.4 确认无误后执行真实迁移

```bash
python tools/migrate_ckpts_to_runs.py --execute
```

迁移后验证：
```bash
# runs/ 结构
find runs -type f -name "*.yaml" | sort   # profile + run_info 应都在
find runs -name "*.ckpt" | sort           # ckpt 应在 runs/{id}/train/checkpoints/

# 旧 checkpoints/ 应已空（可删）
find checkpoints -type f 2>/dev/null   # 应无输出
rmdir checkpoints 2>/dev/null && echo "空 checkpoints/ 已删" || echo "checkpoints/ 仍有内容（手动检查）"

# best ckpt 自动选择验证
python -c "
import sys; sys.path.insert(0,'.')
from tools.config_utils import get_best_ckpt
import os
for d in os.listdir('runs'):
    if os.path.isdir(f'runs/{d}'):
        b = get_best_ckpt(d)
        print(f'{d}: {b}')
        assert b and os.path.exists(b), f'best ckpt 不存在!'
print('✓ 所有 run 的 best ckpt 可正确选择')
"
```

### 2.5 手动迁移非标准目录（仅当 2.3 有无法识别的目录）

如果服务器有自定义命名的权重目录，迁移脚本识别不了，手动处理：

```bash
# 示例: 服务器有 checkpoints/my_completion_exp/ 里的 ckpt
# 先看它的 run_info 确认 task/dataset
cat checkpoints/my_completion_exp/run_info_*.yaml | grep -E "script:|config_source:"

# 假设确认是 completion × spacenet, 时间戳 20260620_100000
TASK=completion
DATASET=spacenet
TS=20260620_100000     # 从 run_info 文件名或内容提取
RUN_ID=${TASK}_${DATASET}_${TS}

mkdir -p runs/${RUN_ID}/train/checkpoints
mv checkpoints/my_completion_exp/*.ckpt runs/${RUN_ID}/train/checkpoints/
mv checkpoints/my_completion_exp/run_info_*.yaml runs/${RUN_ID}/train/

# 补 profile (用 migrate 脚本里的逻辑, 或手动写最小 profile)
python -c "
import sys; sys.path.insert(0,'.')
from tools.config_utils import write_profile, mark_step_done, save_config_snapshot
from tools.run_info import _try_get_git
import os
rid = '${RUN_ID}'
# 读 run_info 拿 config_source
import yaml
ri = [f for f in os.listdir(f'runs/{rid}/train') if f.startswith('run_info')][0]
with open(f'runs/{rid}/train/{ri}') as f: d = yaml.safe_load(f)
cs = d.get('config_source')
profile = {
    'run_id': rid, 'task': '${TASK}', 'dataset': '${DATASET}',
    'created_at': '${TS}', 'migrated_from': 'checkpoints/my_completion_exp',
    'config_source': cs, 'config_snapshot': save_config_snapshot(cs, rid),
    'steps_requested': ['train','infer','eval'],
    'git': _try_get_git('.'), 'step_status': {'train': 'done (migrated)'},
    'note': '手动迁移的非标准目录',
}
write_profile(rid, profile)
mark_step_done(rid, 'train')
print('手动迁移完成')
"
```

## 三、适配 .gitignore（服务器可能有自己的）

服务器的 `.gitignore` 可能和本地不同。目标是确保 `runs/` 的**溯源文件进 git，大产物不进 git**。

### 3.1 检查现有 .gitignore

```bash
grep -nE "runs|ckpt|checkpoints|save" .gitignore
```

### 3.2 确保这些规则存在（缺则补）

`runs/` 需要的规则（如果 `*.ckpt` 已全局忽略，ckpt 自动不进 git）：

```gitignore
# runs/ 统一实验目录: 保留溯源 yaml/json, 忽略大产物
runs/*/*/csv/
runs/*/infer/mask/
runs/*/infer/viz/
runs/*/infer/graph/
runs/*/eval/apls/
runs/*/eval/topo/
runs/*/train/csv/
!runs/**/*.yaml
!runs/**/*.json
```

如果服务器的 .gitignore 没有 `runs/` 相关规则，**追加**上面的内容（不要删除服务器已有的其他规则）。

### 3.3 验证 gitignore 行为正确

```bash
# 确认 ckpt 被忽略
git check-ignore runs/*/train/checkpoints/*.ckpt 2>/dev/null && echo "✓ ckpt 被忽略"

# 确认 profile/run_info 会被跟踪
git add -n runs/ 2>&1 | grep -E "profile.yaml|run_info" | head
# 应看到 profile.yaml 和 run_info_*.yaml
```

### 3.4 处理服务器自己的 ignore 规则

服务器可能在 `.gitignore` 里有自己的规则（比如忽略某些数据集、某些日志）。**不要动这些**，只补 `runs/` 相关的。如果服务器的规则和 `runs/` 冲突（比如 `runs/` 被整体忽略），需要加 `!runs/**/*.yaml` 例外来放行溯源文件。

## 四、端到端验证（必做）

迁移和 .gitignore 适配后，必须验证编排链路在服务器上能跑通。

### 4.1 验证 GPU 可用

```bash
python -c "import torch; print('CUDA:', torch.cuda.is_available(), '设备数:', torch.cuda.device_count())"
nvidia-smi --query-gpu=index,name,memory.free --format=csv 2>/dev/null | head
```

记录可用 GPU id。服务器的 GPU id 可能和本地不同。

### 4.2 dry-run 验证命令构造（不耗时）

```bash
# 四个任务组合的 config 路径都应正确
for combo in "extraction spacenet" "extraction didi_xian" "completion spacenet" "completion didi_xian"; do
  set -- $combo
  python run.py --task $1 --dataset $2 --steps train --gpus 0 --dry-run 2>&1 | grep -oE "config/[^ ]*\.yaml" | head -1
done
```

确认每个 config 文件真实存在：
```bash
python -c "
import sys; sys.path.insert(0,'.')
from tools.registry import default_config_for
import os
for t in ['extraction','completion']:
    for d in ['spacenet','didi_xian']:
        c = default_config_for(t,d)
        print(('✓' if os.path.exists(c) else '✗'), t, d, c)
"
```

若有 ✗（config 不存在）：服务器的 config 文件名可能不同。检查 `ls config/` 并用 `--config` 显式指定，或修正 `tools/registry.py` 的 `default_config_for`。

### 4.3 真实推理冒烟测试（用迁移的 ckpt）

选一个迁移来的 run，跑单步 infer 验证整条链路：

```bash
# 找一个迁移的 run_id
RUN_ID=$(ls runs/ | grep completion_spacenet | head -1)
echo "用 $RUN_ID 测试"

# 真实推理 (GPU, 只验证能跑通, 完整推理会处理整个 test 集)
python run.py --task completion --dataset spacenet \
    --run-id $RUN_ID --steps infer --resume-run --checkpoint auto --gpus 0 2>&1 | tail -20
```

**预期看到**：
- `续跑 $RUN_ID, 已有 profile 加载`
- `▶ [infer] ... --checkpoint runs/.../best.ckpt --run-root runs/...`
- `Loaded known graph with N nodes`（completion 模式）
- 开始 `Processing <img_id>`

**若报错 `expected scalar type Float but found Double`**：说明服务器的 inferencer 还有 dtype bug。检查：
```bash
grep "batch_points = torch.tensor" engine/inferencer_completion.py
# 应该是 dtype=torch.float32。若没有 → 该 commit 没合并, git pull
```

**若报错 ckpt 找不到**：`best_ckpt.txt` 指向的路径可能因迁移时机器不同而失效。重新选：
```bash
python -c "
import sys; sys.path.insert(0,'.')
from tools.config_utils import select_best_ckpt
print(select_best_ckpt('$RUN_ID'))
"
```

### 4.4 验证产物结构

冒烟测试跑通几张图后（可 Ctrl+C 中断，不必跑完整个 test 集），检查：

```bash
find runs/$RUN_ID/infer -type f | head
# 应有 graph/ mask/ viz/ 下的文件, config.yaml, run_info.yaml
```

## 八、故障排查

| 症状 | 排查 |
|------|------|
| `--checkpoint auto` 选不到 ckpt | `runs/{id}/train/checkpoints/` 下是否有 `.ckpt`？运行 `select_best_ckpt` 重新选。文件名需含 `val_loss=` 才能被识别为 best |
| `expected Float but found Double` | inferencer 的 `batch_points` 未指定 float32，`git pull` 拿 dtype 修复 |
| config 不存在 | `ls config/` 看实际文件名，用 `--config` 显式指定，或修 `tools/registry.py` 的 `default_config_for` |
| dataset 名报错 | 注册表 key 是 `spacenet`/`didi_xian`/`cityscale`，不是 `xian`。`grep -r "xian" tools/registry.py` 确认 |
| 推理 OOM | `--gpus` 选空闲 GPU；或减小 config 的 `INFER_BATCH_SIZE` |
| `--resume-run` 找不到 profile | `runs/{id}/profile.yaml` 必须存在。迁移没覆盖的 run 需手动补 profile |
| ckpt 被误提交到 git | `.gitignore` 缺 `*.ckpt`；`git rm --cached <ckpt>` 移除后补 `.gitignore` |
| 迁移脚本识别不了目录 | 目录名不符 `samroad_{variant}_{dataset}` 规则，按 2.5 手动迁移 |

## 九、给服务器 Claude 的操作约束

1. **绝不照搬本手册或本地的时间戳/run_id**——服务器的时间戳由其 run_info 决定。
2. **每一步先探测再操作**：迁移前 dry-run，提交前 `git status` 确认无 ckpt 泄漏。
3. **不动服务器自己的 .gitignore 规则**，只补 `runs/` 相关的。
4. **不删除任何 ckpt 文件**，只移动到 `runs/` 下。ckpt 是昂贵产物。
5. **冒烟测试可中断**，不必跑完整个 test 集，验证链路通即可。
6. **push 前必查**：`git status -s | grep ckpt` 必须为空。
7. **遇到本手册未覆盖的服务器特殊情况**（自定义脚本、非标准 config 命名等），先理解其逻辑再适配，不要强行套用。
